# Day 1 - NanoDump Lab Summary

This document provides a standalone summary of the workflow across adversary emulation, telemetry collection, and threat hunting.

## Scope

The sections below can be read independently as a complete lab sequence:
- Adversary emulation and EDR evasion.
- Telemetry generation and pipeline architecture.
- Threat hunting and cryptographic verification.

## Adversary Emulation & EDR Evasion

### Objective
Validate detection coverage for MITRE ATT&CK T1003.001, OS Credential Dumping: LSASS Memory, using realistic evasion-capable tooling.

### Work Completed
- Deployed the Atomic Red Team execution framework via administrative PowerShell.
- Executed simulated LSASS-dumping activity.
- When native tooling was blocked by Windows Defender or AMSI, pivoted to `nanodump.x64.exe` as the fallback execution path.

### Notes
- The intent was to test behavioral detection rather than only basic antivirus signatures.
- The fallback technique was used to better emulate an advanced attacker path.

## Telemetry Generation & Pipeline Architecture

### Objective
Generate high-fidelity telemetry on the Windows endpoint and route it into the Elasticsearch stack.

### Work Completed
- Installed Sysinternals Sysmon using the SwiftOnSecurity XML configuration.
- Updated `winlogbeat.yml` to ingest standard event channels and `Microsoft-Windows-Sysmon/Operational`.
- Restarted the Winlogbeat service after Sysmon was installed so the Sysmon channel was correctly hooked.

### Notes
- Sysmon improved visibility into process creation, memory access, and file creation.
- Restarting Winlogbeat after Sysmon installation was necessary to avoid missing the new registry-backed channel.

## Threat Hunting & Cryptographic Verification

### Objective
Reconstruct the attack chain in Kibana and separate real malicious activity from background noise.

### Work Completed
- Queried Sysmon Event IDs 1, 10, and 11 to reconstruct process lineage.
- Matched activity by timing to separate the attack window from unrelated later system activity.
- Used `winlog.event_data.OriginalFileName` and hash-based validation to avoid relying on a renamed binary path.

### Notes
- Filename-only hunting was treated as insufficient because attackers can rename binaries to common-looking names.
- The immutable metadata and cryptographic indicators provided a better indicator of compromise.
